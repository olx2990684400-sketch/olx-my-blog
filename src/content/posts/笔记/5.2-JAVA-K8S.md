# Java 开发者的 K8S 进阶之路：从容器化入门到企业级实践

> 一篇面向 Java 后端开发者的 Kubernetes 实战指南。从"为什么我的 Spring Boot 应用需要 K8S"这个问题出发，逐步覆盖核心概念、部署实战、配置管理、生产级方案，最终落地到企业级 CI/CD 集成。读完本文，你能独立完成一个 Java 应用的容器化部署，并理解企业级 K8S 集群的关键设计决策。

## 引子

场景：线上一个 Spring Boot 服务挂了，运维手动重启，10 分钟后恢复。凌晨 3 点又挂了，值班运维没看到告警，服务中断 2 小时。事后复盘——OOM、线程池耗尽、流量突增，每个原因都指向同一个问题：**单机部署没有自愈能力**。

这不是个例。Java 应用在生产环境面临三个现实问题：

1. **单点故障**：进程崩溃后无人拉起，靠人工介入恢复，MTTR（平均恢复时间）以小时计
2. **扩缩容迟钝**：大促流量来了，手动加机器，流量走了，机器空转烧钱
3. **环境漂移**：开发、测试、生产三套环境配置不一致，"在我机器上是好的"成为日常

Kubernetes 解决的不是某一个技术点，而是提供了一套声明式的容器编排能力：你告诉集群"我要什么状态"，集群负责维持这个状态。进程挂了自动重启，流量大了自动扩容，环境通过镜像固化保证一致性。

本文按三个阶段组织：初学阶段掌握核心对象和基本部署，进阶阶段搞定配置和存储，企业级阶段落地生产方案。

## 背景知识

### 容器化：K8S 的前提

K8S 编排的对象是容器，不是进程。理解 K8S 之前，先理解容器解决了什么问题。

传统部署方式中，Java 应用直接跑在物理机或虚拟机上，依赖 JDK 版本、系统库、环境变量等宿主机状态。容器（Docker）把这些依赖打包进镜像，保证"构建一次，到处运行"。

```dockerfile
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

这条 Dockerfile 做了一件事：把 JDK 21 + 应用 JAR 打包成一个不可变镜像。无论在开发机还是生产集群，镜像内容完全一致。

### K8S 核心架构

[配图：K8S 集群架构图，展示 Master 节点（API Server、Scheduler、Controller Manager、etcd）与 Worker 节点（kubelet、kube-proxy、容器运行时）的关系]

K8S 集群由两类节点组成：

| 组件 | 职责 | 类比 |
|------|------|------|
| API Server | 集群入口，所有操作经此 | 前台接待 |
| etcd | 持久化存储集群状态 | 数据库 |
| Scheduler | 决定 Pod 运行在哪个节点 | 调度员 |
| Controller Manager | 维护期望状态（如副本数） | 巡检员 |
| kubelet | 节点上的代理，管理 Pod 生命周期 | 现场工人 |
| kube-proxy | 节点上的网络代理，实现 Service | 交换机 |

关键认知：**K8S 是声明式系统**。你提交一个 YAML 声明"我要 3 个副本的 Nginx"，Controller Manager 持续对比期望状态和实际状态，发现偏差就自动修正。这和脚本式运维（"先启动 A，再启动 B"）有本质区别。

## 核心内容

### 第一阶段：初学——理解核心对象

#### Pod：最小调度单元

Pod 是 K8S 中最小的部署单元，不是容器。一个 Pod 可以包含一个或多个容器，它们共享网络命名空间（同一 IP）和存储卷。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-registry/app:1.0.0
      ports:
        - containerPort: 8080
      resources:
        requests:
          memory: "512Mi"
          cpu: "250m"
        limits:
          memory: "1Gi"
          cpu: "500m"
```

Java 开发者需要特别关注 `resources` 字段。JVM 默认会尝试使用宿主机的全部内存，在容器环境中这会导致 OOM Killed。必须设置 `limits` 限制容器内存上限，同时 JVM 启动参数配合：

```bash
java -XX:MaxRAMPercentage=75.0 -jar app.jar
```

`-XX:MaxRAMPercentage=75.0` 让 JVM 堆内存取容器内存限制的 75%，而非宿主机内存。这是 Java 应用容器化的第一个坑。

#### Deployment：声明式副本管理

直接创建 Pod 没有自愈能力——Pod 所在节点故障，Pod 就丢失了。Deployment 在 Pod 之上增加了一层管理：维护指定数量的 Pod 副本，支持滚动更新和回滚。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: app
          image: my-registry/app:1.0.0
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 60
            periodSeconds: 15
          resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
            limits:
              memory: "1Gi"
              cpu: "500m"
```

Java 开发者注意两个 Probe 配置：

- **readinessProbe**：Pod 就绪后才接收流量。Spring Boot Actuator 提供 `/actuator/health/readiness` 端点，应用启动完成且依赖（数据库、Redis）连通后才返回 200
- **livenessProbe**：Pod 存活检查，失败则重启容器。`initialDelaySeconds` 必须大于 JVM 启动时间，否则应用还没启动完就被杀掉，陷入 CrashLoopBackOff 循环

滚动更新策略：

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

`maxUnavailable: 0` 保证更新期间旧 Pod 全部可用，`maxSurge: 1` 允许临时多出一个 Pod。这是 Java 应用平滑发布的最低要求——JVM 启动慢（通常 15-30 秒），必须保证旧实例在新的就绪前不退出。

#### Service：稳定的访问入口

Pod 的 IP 是临时的，Pod 重建后 IP 变化。Service 为一组 Pod 提供固定 IP 和 DNS 名称。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP
```

`type: ClusterIP` 分配集群内部 IP，集群内通过 `my-app-svc.default.svc.cluster.local` 访问。Service 通过 `selector` 匹配 Pod 的 label，自动将流量负载均衡到健康的 Pod。

三种 Service 类型对比：

| 类型 | 访问范围 | 适用场景 |
|------|----------|----------|
| ClusterIP | 集群内部 | 微服务间调用 |
| NodePort | 集群外部通过节点端口 | 测试环境、非 HTTP 服务 |
| LoadBalancer | 云厂商提供外部 LB | 生产环境对外暴露 |

### 第二阶段：进阶——配置与存储

#### ConfigMap 和 Secret：配置外置

Java 应用的配置（数据库地址、线程池大小、日志级别）不应该硬编码在镜像里。ConfigMap 存储非敏感配置，Secret 存储敏感信息（密码、Token）。

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  SPRING_PROFILES_ACTIVE: "prod"
  LOG_LEVEL: "INFO"
  DB_HOST: "mysql.prod.svc.cluster.local"
  DB_PORT: "3306"
---
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DB_PASSWORD: cHJvZC1wYXNzd29yZA==
```

在 Deployment 中引用：

```yaml
env:
  - name: SPRING_PROFILES_ACTIVE
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: SPRING_PROFILES_ACTIVE
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: app-secret
        key: DB_PASSWORD
```

**注意**：Secret 的 `data` 字段是 Base64 编码，不是加密。生产环境中 Secret 需要配合加密方案（如 Sealed Secrets、Vault）使用。Base64 编码只是避免明文出现在 YAML 中，任何有集群读权限的人都能解码。

#### PV 和 PVC：持久化存储

容器文件系统是临时的，容器重启后数据丢失。对于数据库、日志等需要持久化的数据，使用 PersistentVolume（PV）和 PersistentVolumeClaim（PVC）。

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-data
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi
```

在 Pod 中挂载：

```yaml
volumes:
  - name: data
    persistentVolumeClaim:
      claimName: app-data
volumeMounts:
  - name: data
    mountPath: /app/data
```

Java 应用中，日志目录、上传文件目录、嵌入式数据库目录是需要持久化的典型场景。`accessModes` 中 `ReadWriteOnce` 表示只允许一个节点挂载读写，`ReadWriteMany` 允许多节点同时读写（需要存储后端支持，如 NFS）。

#### Ingress：HTTP 路由入口

Service 的 ClusterIP 只在集群内可达，NodePort 端口范围有限（30000-32767），LoadBalancer 每个服务独占一个外部 IP。Ingress 提供基于域名和路径的 HTTP 路由，一个入口点分发到多个 Service。

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: my-app-svc
                port:
                  number: 80
          - path: /admin
            pathType: Prefix
            backend:
              service:
                name: admin-svc
                port:
                  number: 80
  tls:
    - hosts:
        - api.example.com
      secretName: api-tls
```

Ingress 是生产环境对外暴露 HTTP 服务的标准方式。配合 cert-manager 可以自动签发和续期 TLS 证书。

### 第三阶段：企业级——生产方案

#### Namespace：多租户隔离

企业级集群通常按环境（dev/staging/prod）或团队划分 Namespace，实现资源配额和网络策略的隔离。

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: prod
  labels:
    name: prod
```

配合 ResourceQuota 限制资源使用：

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: prod-quota
  namespace: prod
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    pods: "50"
```

没有 ResourceQuota 的 Namespace，任何一个团队都能占满集群资源。这是生产集群的必修课。

#### RBAC：权限控制

默认情况下，ServiceAccount 在 Namespace 内有完整权限。生产环境必须按最小权限原则配置 RBAC。

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: app-deployer
  namespace: prod
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "update", "patch"]
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-deployer-binding
  namespace: prod
subjects:
  - kind: ServiceAccount
    name: ci-sa
roleRef:
  kind: Role
  name: app-deployer
  apiGroup: rbac.authorization.k8s.io
```

这条规则只允许 CI 服务账号在 prod 命名空间内更新 Deployment 和查看 Pod 日志，不能删除资源、不能访问 Secret。

#### HPA：自动扩缩容

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Percent
          value: 50
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
```

关键参数说明：

- `stabilizationWindowSeconds`：扩容等待 60 秒确认趋势，缩容等待 300 秒避免抖动。Java 应用启动慢，缩容窗口必须足够长，否则刚缩掉的 Pod 马上又要扩回来
- `scaleDown.policy`：每次最多缩 10%，避免流量突降时一次性缩掉过多实例

**HPA 的前提**：Pod 必须设置 `resources.requests`，否则 HPA 无法计算利用率百分比。这是最常见的 HPA 不生效原因。

#### Helm：包管理

当应用部署涉及 Deployment、Service、ConfigMap、Secret、Ingress、HPA 等多个 YAML 时，手动管理效率低且易出错。Helm 把这些 YAML 模板化，通过 `values.yaml` 参数化配置，实现一套模板多环境复用。

项目结构：

```
my-app-chart/
├── Chart.yaml
├── values.yaml
├── templates/
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── configmap.yaml
│   ├── secret.yaml
│   ├── ingress.yaml
│   ├── hpa.yaml
│   └── _helpers.tpl
```

`values.yaml` 示例：

```yaml
replicaCount: 3
image:
  repository: my-registry/app
  tag: "1.0.0"
  pullPolicy: IfNotPresent
resources:
  requests:
    memory: "512Mi"
    cpu: "250m"
  limits:
    memory: "1Gi"
    cpu: "500m"
ingress:
  enabled: true
  host: api.example.com
  tls: true
hpa:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilization: 70
```

部署命令：

```bash
helm upgrade --install my-app ./my-app-chart \
  -n prod \
  -f values-prod.yaml
```

`--install` 表示不存在则安装，存在则升级。`-f values-prod.yaml` 覆盖默认值。Helm 的价值在于：环境差异只体现在 values 文件中，模板逻辑统一维护。

## 实战演示

### 端到端：Spring Boot 应用部署到 K8S

以一个 Spring Boot 3 应用为例，从代码到 K8S 集群运行。

**第一步：构建镜像**

```bash
mvn clean package -DskipTests
docker build -t my-registry/app:1.0.0 .
docker push my-registry/app:1.0.0
```

Dockerfile：

```dockerfile
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", \
  "-XX:MaxRAMPercentage=75.0", \
  "-XX:+UseG1GC", \
  "-Djava.security.egd=file:/dev/./urandom", \
  "-jar", "app.jar"]
```

`-Djava.security.egd=file:/dev/./urandom` 解决 Linux 环境下 SecureRandom 初始化阻塞问题，Spring Boot 应用在 K8S 中启动慢的常见原因之一。

**第二步：编写部署 YAML**

将前面各阶段的 Deployment、Service、ConfigMap、Secret、Ingress、HPA 组合到一个目录：

```
k8s/
├── deployment.yaml
├── service.yaml
├── configmap.yaml
├── secret.yaml
├── ingress.yaml
└── hpa.yaml
```

**第三步：部署到集群**

```bash
kubectl create namespace prod
kubectl apply -f k8s/ -n prod
```

**第四步：验证**

```bash
kubectl get pods -n prod
kubectl get svc -n prod
kubectl get ingress -n prod
kubectl logs -f deployment/my-app -n prod
```

**第五步：滚动更新**

```bash
kubectl set image deployment/my-app app=my-registry/app:1.1.0 -n prod
kubectl rollout status deployment/my-app -n prod
```

回滚：

```bash
kubectl rollout undo deployment/my-app -n prod
kubectl rollout history deployment/my-app -n prod
```

### CI/CD 集成

企业级部署不应手动执行 `kubectl apply`。以下是一个 GitLab CI 流水线示例：

```yaml
stages:
  - build
  - deploy

build:
  stage: build
  script:
    - mvn clean package -DskipTests
    - docker build -t $REGISTRY/app:$CI_COMMIT_SHORT_SHA .
    - docker push $REGISTRY/app:$CI_COMMIT_SHORT_SHA
  only:
    - main

deploy-prod:
  stage: deploy
  image: bitnami/kubectl
  script:
    - kubectl config use-context prod
    - helm upgrade --install app ./chart
        --set image.tag=$CI_COMMIT_SHORT_SHA
        -n prod
        -f values-prod.yaml
  only:
    - main
  when: manual
```

关键设计决策：

- 镜像 Tag 使用 Git Commit SHA，保证每次部署可追溯
- 生产部署 `when: manual`，必须人工确认触发
- 使用 Helm 部署而非 `kubectl apply`，便于版本管理和回滚

## 踩坑点 & 注意事项

### JVM 内存设置与容器 OOM Killed

**现象**：容器启动后一段时间被 OOM Killed，`kubectl describe pod` 显示 `Last State: Terminated, Reason: OOMKilled`。

**原因**：JVM 默认根据宿主机内存计算堆大小，而非容器内存限制。在 64GB 宿主机上运行 `limits.memory: 1Gi` 的容器，JVM 可能尝试分配 16GB 堆内存，远超容器限制。

**解决方案**：

```bash
# 方案 1：使用百分比（推荐）
-XX:MaxRAMPercentage=75.0

# 方案 2：直接指定最大堆
-Xmx768m
```

方案 1 更灵活，容器 limits 变化时不需要改启动参数。75% 留 25% 给元空间、线程栈、堆外内存。

### readinessProbe 导致启动超时

**现象**：Pod 反复重启，状态为 `CrashLoopBackOff`，`kubectl logs` 显示应用正常启动但被杀。

**原因**：`readinessProbe` 的 `initialDelaySeconds` 设置过小，应用还没完成 Spring 上下文初始化就被判定为不健康，流量打不进来，`livenessProbe` 超时后重启容器。

**解决方案**：

```yaml
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 90
  periodSeconds: 15
  failureThreshold: 3
```

`livenessProbe.initialDelaySeconds` 必须大于应用最慢启动时间。Spring Boot 应用连接数据库、初始化缓存后才能响应健康检查，冷启动通常需要 30-60 秒。生产环境建议 liveness 的 `initialDelaySeconds` 设为 90 秒以上。

### ConfigMap 更新后应用不生效

**现象**：修改 ConfigMap 后，Pod 内环境变量没有变化。

**原因**：ConfigMap 作为环境变量注入时，更新 ConfigMap 不会触发 Pod 重启。环境变量在 Pod 创建时写入，运行期间不会动态更新。

**解决方案**：

1. 手动触发滚动更新：`kubectl rollout restart deployment/my-app -n prod`
2. 使用 Spring Cloud Kubernetes Config，配合 `spring.cloud.kubernetes.config.enableApi=true` 实现运行时刷新
3. 在 CI/CD 流水线中，ConfigMap 变更自动触发 Deployment 重新部署

### HPA 不生效

**现象**：CPU 利用率超过阈值，但副本数没有增加。

**常见原因排查**：

1. **未设置 `resources.requests`**：HPA 计算利用率 = 实际使用 / requests，没有 requests 则无法计算
2. **Metrics Server 未安装**：HPA 依赖 Metrics Server 采集指标，`kubectl top pods` 验证是否可用
3. **stabilizationWindowSeconds 过大**：扩容窗口内指标波动，HPA 等待确认

```bash
kubectl get pods -n kube-system | grep metrics-server
kubectl top pods -n prod
kubectl describe hpa my-app-hpa -n prod
```

### 日志收集

容器日志默认存储在容器文件系统中，容器销毁后丢失。生产环境必须集中收集日志。

方案对比：

| 方案 | 侵入性 | 实时性 | 适用规模 |
|------|--------|--------|----------|
| Sidecar（Filebeat/Fluentd） | 高，每个 Pod 多一个容器 | 高 | 中小规模 |
| DaemonSet（Fluent Bit） | 低，节点级采集 | 高 | 大规模 |
| 应用直写（Logstash TCP） | 高，改应用代码 | 最高 | 不推荐 |

推荐 DaemonSet 方案：每个节点运行一个 Fluent Bit Pod，采集 `/var/log/containers/` 下所有容器日志，发送到 Elasticsearch/Loki。应用代码无需修改，日志输出到 stdout/stderr 即可。

Spring Boot 日志输出到 stdout 的配置：

```yaml
logging:
  pattern:
    console: '{"time":"%d","level":"%p","traceId":"%X{traceId}","msg":"%m"}%n'
```

JSON 格式便于日志平台解析和检索，`traceId` 支持分布式链路追踪。

## 性能/对比数据

### Java 应用启动优化

| 优化手段 | 冷启动时间 | 说明 |
|----------|-----------|------|
| 无优化 | 25-40s | 默认 JVM 参数 |
| +MaxRAMPercentage | 20-30s | 减少堆内存浪费 |
| +UseG1GC | 18-28s | G1 比 Serial/Parallel 更适合容器 |
| +TieredStopAtLevel=1 | 12-18s | 跳过 JIT 编译，牺牲峰值性能换启动速度 |
| GraalVM Native Image | 0.05-0.5s | AOT 编译，启动极快但构建复杂 |

TieredStopAtLevel=1 适合 Serverless/按需启动场景，长期运行的服务不建议使用，峰值吞吐量下降约 30%。

### K8S 部署方式对比

| 方案 | 管理复杂度 | 适用规模 | 成本 |
|------|-----------|---------|------|
| kubectl apply | 低 | < 10 个服务 | 无 |
| Kustomize | 中 | 10-50 个服务 | 无 |
| Helm | 中 | 50+ 个服务 | 无 |
| ArgoCD + Helm | 高 | 企业级 | 需要额外运维 |

Kustomize 适合简单的环境差异覆盖（dev/staging/prod），Helm 适合需要模板逻辑的复杂应用。ArgoCD 在 Helm 之上增加 GitOps 能力——Git 仓库是唯一事实来源，集群状态自动同步。

## 部署清单

Java 应用上 K8S 的最小必要配置清单：

| 配置项 | 是否必须 | 说明 |
|--------|---------|------|
| resources.requests/limits | 必须 | 不设则无法调度和 HPA |
| readinessProbe | 必须 | 防止流量打到未就绪的实例 |
| livenessProbe | 必须 | 进程死锁时自动重启 |
| -XX:MaxRAMPercentage | 必须 | 防止 JVM 超出容器内存限制 |
| ConfigMap/Secret | 必须 | 配置外置，镜像与环境解耦 |
| Ingress | 生产必须 | 对外暴露 HTTP 服务 |
| HPA | 生产推荐 | 自动扩缩容 |
| ResourceQuota | 生产必须 | 防止资源抢占 |
| RBAC | 生产必须 | 最小权限原则 |
| 日志采集 | 生产必须 | 容器日志不持久化 |

## 总结

K8S 的核心价值是声明式状态管理：你定义期望状态，集群负责维持。对 Java 开发者而言，关键转变在于三点：

1. **从进程思维到容器思维**：JVM 不是跑在裸机上，内存、CPU、网络都受容器限制，必须显式配置
2. **从命令式到声明式**：不写"启动 3 个实例"的脚本，写"期望 3 个副本"的 YAML，让 K8S 维持状态
3. **从单机到分布式**：应用必须无状态化，会话、缓存、文件存储都外置到独立服务

初学阶段掌握 Pod、Deployment、Service 三个核心对象，就能完成基本部署。进阶阶段搞定 ConfigMap、Secret、PV/PVC、Ingress，实现配置外置和持久化。企业级阶段落地 Helm、RBAC、HPA、日志采集、CI/CD 集成，才是生产可用的 K8S 方案。

K8S 不是银弹。对于 3 个以内的服务、不需要自动扩缩容的场景，Docker Compose 足够。K8S 的复杂度是真实存在的——etcd 运维、网络策略、证书管理，每一项都需要投入学习成本。选择 K8S 的前提是：你的服务规模和运维需求已经超出了简单部署工具的能力边界。
